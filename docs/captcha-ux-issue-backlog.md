# 验证码体验与安全问题清单

本文档记录 Demo 验收中确认的问题、修复进展和验收状态，用于后续修复排期与回归检查。

记录日期：2026-06-22

## 状态约定

| 状态 | 含义 |

| 已记录 | 问题已确认，尚未开始修复 |

## P0 / P1 问题

### UX-001 手势描绘校验过松

- 验证码类型：`GESTURE`
- 优先级：P0
- 状态：待人工验收
- 现象：用户只画一条接近水平的轨迹，没有沿提示图形描绘，也能通过验证并签发 ticket。
- 影响：安全风险高。脚本可能只要从起点拖到终点附近就能通过。
- 期望：校验必须覆盖路径形态，至少检查关键拐点顺序、路径覆盖度、偏离距离、方向、轨迹断点和耗时分布。
- 验收点：
  - 明显直线穿过提示图形时不能通过。
  - 没有经过主要拐点时不能通过。
  - 正常按提示描绘时可以稳定通过，柔和弧线、开口环和回钩类路径不能因为用户采样点稀疏或轻微偏移被误杀。
  - Browser smoke 需要加入“错误直线应失败”的反向用例。
- 当前进展：服务端已将手势描绘校验从逐点贴合调整为按路径长度重采样后的整体曲线相似度校验，并结合路径长度、弯曲度和推进顺序拦截起终点直线捷径；单元测试覆盖各手势轨迹族的自然曲线通过和直线捷径失败。弧线/开口环手感仍需人工在 Demo 中复验。

### UX-002 手势描绘题目轨迹固定

- 验证码类型：`GESTURE`
- 优先级：P0
- 状态：已自动验收
- 现象：提示轨迹为固定形状/固定关键点，刷新后的变化不足。
- 影响：安全风险高。即使校验变严格，固定题目仍容易被脚本复用固定路径。
- 期望：每次 challenge 动态生成轨迹，轨迹族本身必须不同，不能只做端点、控制点、缩放、平移或局部扰动；点位不同但整体仍是同一种折线不具备有效防护价值，破解脚本无需改代码即可继续复用。
- 体验约束：动态轨迹不能过难、过乱、锯齿化、自交严重或超出移动端可操作范围；同族变化也必须是自然控制点、曲率和走势变化，不能靠人类难以绘制的折线抖动制造差异。
- 验收点：
  - 连续刷新至少出现明显不同且真人可描绘的路径族，例如柔和波浪、拱形、S 形、回钩、开口环等，而不是同一类折线的抖动、翻转或锯齿化。
  - 即使抽到同一轨迹族，也要通过控制点、振幅、曲率和尾部走势变化生成不同曲线，不能复用固定模板。
  - challenge payload 不暴露答案轨迹以外的校验规则和容差。
  - 正常用户仍可在合理时间内完成。
- 当前进展：旧的多模板随机扰动已判定不合格，因为肉眼观感仍像同一类轨迹；生成器已改为柔和波浪、拱形、S 形、回钩、开口环等真人可描绘的多轨迹族，服务端刷新会避开上一题同族轨迹，同一轨迹族内部也会随机化控制点和曲率，并增加结构签名测试，避免只用点位变化冒充动态轨迹。2026-06-22 已进一步收敛回钩和开口环，避免路径自交导致弧线类手势被校验器误判。

### UX-003 滑动曲线交互形态错误

- 验证码类型：`CURVE` / `CURVE_V2` / `CURVE_V3`
- 优先级：P0
- 状态：已修复，待人工验收
- 现象：早期实现曾把滑动曲线做成用户按提示绘制整条曲线，本质上和 `GESTURE` 没有区别；随后又一度误做成竖向片段还原。
- 影响：功能对齐错误。三种滑动曲线没有和手势描绘拉开差异，不符合目标参考体验。
- 期望：`GESTURE` 保留为自由描绘类；`CURVE` / `CURVE_V2` / `CURVE_V3` 参考 Tianai 在线体验的滑动曲线形态：背景图上叠加 canvas 曲线层，用户拖动底部滑块让移动曲线与目标曲线重合，而不是画线、沿路径移动点、拖竖向缺口片段，或移动一张前景图片。
- 体验约束：曲线层必须自然、轻量、无答案边框；底部 slider 松手自动验证；目标位置只能通过可见曲线的视觉匹配体现，不能通过 `target_x`、`answer_x` 等 JSON 字段直接泄露；滑动不能简化成整条曲线横向平移。
- Tianai 观察基线：
  - 观察时间：2026-06-23，用户手动刷新目标站点后出现 `滑动曲线V3`。
  - 根容器：`#tianai-captcha`，class 包含 `tianai-captcha-slider`。
  - 文案：`拖动滑块使曲线匹配`。
  - 背景图：`#tianai-captcha-slider-bg-img`，natural size `600x360`，展示尺寸约 `300x180`；图片加载慢时 challenge 会延迟出现。
  - 画布层：`#tianai-captcha-slider-bg-canvas` 与 `#tianai-captcha-curve-bg-canvas`，展示尺寸约 `300x180`。
  - 端点元素：存在 `#tianai-captcha-curve-ball-div-left` 与 `#tianai-captcha-curve-ball-div-right`；端点是否显示由类型和题目状态决定，但端点一旦显示，拖动过程中坐标不随 slider 位移变化，不能把端点做成随滑块移动的元素。
  - 滑块与轨道：存在 `#tianai-captcha-slider-move-btn` 和 `#tianai-captcha-slider-move-track-mask`；拖动后按钮使用 `transform: translate(...)`，轨道 mask 宽度随滑动距离变化。
  - 滑动方式：滑块驱动按钮位移、轨道 mask 宽度和 canvas 曲线状态；不得把滑块值直接实现为整条曲线 `translateX(value)`，也不得根据滑块值重算左右端点坐标。
  - 关键否定项：未观察到独立的 `slider-move-img` 或 `.curve-piece` 前景图片；不应实现成图片 piece、竖向切片、自由绘制路径、端点跟随滑块移动，或两条自创彩色曲线。
  - 异常反馈：一次拖动后出现过 `验证码被黑洞吸走了！`，说明目标实现包含失败/风控提示分支，不只是静态偏移校验。
  - V1 / V2 / V3 归类：三者应共享同一类滑动曲线体验，只做曲线复杂度、素材、扰动和策略差异，不应变成三种完全不同交互。
- 2026-06-23 二次观察补充：
  - 观察产物：截图保存到 `output/tianai-observe-20260623/v3-before.png`、`output/tianai-observe-20260623/v3-after-release.png`；目标站点脚本保存到 `output/tianai-observe-20260623/tac.min.js`、`output/tianai-observe-20260623/index.js`。
  - V3 干净初始态：按钮 `transform=0`，track mask 宽度约 `2px`，背景图 `600x360 -> 300x180`，两个 canvas 均为 `300x180`；一次题目中端点初始隐藏。
  - V3 拖动中未松手：拖到 `120px` 时按钮 `transform=120px`、mask `width=120px`，左端点 `left:11.5px; top:18.5px`，右端点 `left:232.5px; top:6.5px`；继续拖到 `170px` 时按钮和 mask 变为 `170px`，两个端点 style 和 rect 保持不变。
  - V3 松手失败态：出现 `验证码被黑洞吸走了！`；失败动画会让根容器出现横向 transform，这属于失败反馈，不是正常拖动时移动整张验证码。
  - V2：同样使用背景图、双 canvas、按钮和 track mask；拖到 `135px` 时按钮和 mask 变为 `135px`，左右端点保持隐藏，失败提示为 `验证失败，请重新尝试!`。
  - V1：端点初始可见；拖到 `135px` 时按钮和 mask 变为 `135px`，左端点保持 `left:52.5px; top:15px`，右端点保持 `left:275px; top:3px`，不随 slider 移动。
  - JS 证据：`tac.min.js` 中存在曲线模板的双 canvas 与端点 div；关键词包含 `loadByCurve1`、`loadByCurve2`、`drawDoubleBezier`、`pathPoints`、`drives`、`doMove`、`#tianai-captcha-slider-move-track-mask` 和两种失败文案，说明曲线渲染围绕 Bezier/path/move 驱动与轨道 mask，而不是图片 piece。
- 验收点：
  - 三种曲线不再使用“自由绘制整条线”的交互。
  - 三种曲线不再使用竖向缺口/红框/目标边缘提示。
  - 松手后自动验证。
  - 错误偏移不能通过，正确曲线匹配后通过。
  - Runtime DOM 必须体现目标结构：背景图、canvas 曲线层、端点元素、滑块按钮和轨道 mask；不得出现 `.curve-piece` 或其他可移动前景图片。
  - 滑动过程中端点坐标必须保持固定或隐藏，不能随 slider 平移；按钮 transform 与 track mask 宽度可以随 slider 变化。
  - 人工验收必须和 Tianai 观察基线逐项对照，自动烟测只能作为回归补充，不能单独判定体验完成。
  - Browser smoke 覆盖曲线匹配成功和错误偏移失败。
- 当前进展：Runtime 已按 Tianai 观察基线重做 `CURVE` / `CURVE_V2` / `CURVE_V3`：根容器、背景图、双 canvas、端点 div、滑块按钮、轨道 mask 与松手自动验证均已接入；拖动时按钮和 mask 随距离变化，端点保持固定或隐藏，曲线通过 drive profile 做形变匹配，不再整体 `translateX`，也不再使用 `.curve-piece`、竖向片段或自由绘制路径。服务端按本次目标偏移校验，并校验提交偏移与 pointer track 末端一致性。Browser smoke 已覆盖三种曲线的错误偏移失败、正确匹配通过和 runtime 渲染；仍需人工对照参考站点确认视觉质感和手感。
- 2026-06-23 体验复查修正：曲线初始动态偏小、肉眼重合偶发失败。已将曲线初始错位改为明确的像素级动态错位，再按本次 answer 反推 drive，避免初始态趋近重叠；`moving_points` 保留浮点，不再取整，确保视觉最佳重合位置和服务端目标值一致；曲线答案容差调整为 10px、轨迹末端一致性容差调整为 13px。单元测试新增平均初始错位、最大错位和视觉目标反推一致性约束。
- 2026-06-23 二次体验澄清：问题不是把三种曲线都追成同一张 Tianai 截图，而是 `CURVE` / `CURVE_V2` / `CURVE_V3` 视觉同质化，用户感知上可以合并成一个。已增加 `visual_style`：`CURVE=single-rope`，`CURVE_V2=dual-noise`，`CURVE_V3=ring-deform`；前端按不同模板渲染，分别对应基础单绳、当前保留的双轨噪声、以及 Tianai V2 观察到的闭合圆环形变。
- 2026-06-23 三次体验澄清：用户指出 Tianai V2 的关键不是普通双轨阴影，而是“闭合圆环被拖动变形、没有显式起点”。当前 `CURVE_V2` 的双轨噪声体验先保留，Tianai V2 圆环效果迁入 `CURVE_V3`，V3 必须隐藏端点、使用闭合路径和周期形变 drive。
- 2026-06-23 四次修正：`CURVE` / `CURVE_V2` / `CURVE_V3` 的目标曲线必须烘进后端返回的 PNG 背景图，并由单元测试检查目标虚影像素存在；前端 profile 仍不得下发目标曲线点。

### UX-004 滑动曲线不能使用固定曲线

- 验证码类型：`CURVE` / `CURVE_V2` / `CURVE_V3`
- 优先级：P0
- 状态：已修复，待人工验收
- 现象：当前曲线图像和匹配位置如果固定或变化不足，脚本容易复用固定偏移。
- 影响：安全风险高。固定缺口位置或固定曲线图像会让脚本容易复用答案。
- 期望：每次 challenge 动态生成曲线 profile、移动初始位置和目标 x，包含控制点、曲率、局部扰动和视觉主题变化；服务端按本次私有偏移校验。
- 体验约束：动态曲线要避免过度弯折、锯齿、交叉和不可理解区域；canvas 曲线层必须包含足够视觉参照，方便用户肉眼对齐。
- 验收点：
  - 连续刷新可以得到不同曲线图像、主题和目标位置。
  - 生成曲线满足最小纹理、边界留白和可对齐性约束。
  - 客户端不暴露答案、目标 x 或容差。
- 当前进展：内置曲线生成器已改为动态背景图 + canvas 曲线 profile，每次 challenge 会生成不同目标曲线、移动曲线、drive 向量和目标偏移；目标曲线只烘进服务端 PNG，不再以 `fixed_points`、`curve_path`、`target_x`、`answer_x`、`split_x` 等结构化字段下发。前端拿到的是可拖动曲线和滑动范围，脚本只能从可见 PNG/Canvas 做视觉识别，不能直接从 JSON/DOM 反推出答案；服务端会同时校验提交偏移、pointer track 末端一致性和轨迹评分，后续防护继续依赖一次性 session、频控、风险策略和训练模型。
- 2026-06-23 体验复查修正：新增单元测试要求曲线 profile 不暴露目标曲线点，且曲线 drive 在目标偏移下的平均初始错位不低于 22px、最大错位不低于 34px，避免动态不足和结构化答案泄露回归。

### UX-005 点选类目标圈泄露识别特征

- 验证码类型：`WORD_IMAGE_CLICK` / `IMAGE_CLICK`
- 优先级：P0
- 状态：已自动验收
- 现象：文字或图标背后存在明显圆形/靶心背景，用于增强可读性，但同时直接暴露目标位置。
- 影响：安全风险高。脚本可以先识别圆形区域，再点击目标位置，绕过文字/图标识别难度。
- 期望：去掉明显目标圈，不暴露“这里可点”的视觉特征。
- 体验约束：去掉目标圈后仍要保证可读性，避免文字堆叠；通过布局留白、字体尺寸、轻量干扰、随机但不重叠的点位解决。
- 验收点：
  - 目标背后不再有稳定圆形/靶心特征。
  - 文字和图标不重叠、不贴边、可读。
  - Browser smoke 仍能按视觉目标点选通过。
- 当前进展：内置点选渲染和资源字体合成已移除目标圆形/靶心承载区域。

### UX-006 移除语序点选类型

- 验证码类型：`WORD_ORDER_IMAGE_CLICK`
- 优先级：P1
- 状态：已自动验收
- 现象：语序点选和文字点选在当前实现里交互与校验几乎相同。
- 影响：增加维护、资源和测试成本，但没有带来独立安全价值。
- 期望：去掉 `WORD_ORDER_IMAGE_CLICK`，不作为独立验证码类型继续实现。
- 兼容建议：如果需要保留历史兼容，可将其作为废弃别名映射到 `WORD_IMAGE_CLICK`，但不在 Demo 和主类型矩阵中展示。
- 验收点：
  - 类型矩阵、前端 Demo、后端生成/校验、资源类型、文档和 browser smoke 同步移除或标记废弃。
  - 外部兼容行为有明确策略。
- 当前进展：Demo、browser smoke、资源类型、随机类型和主类型契约已移除该类型；外部请求会兼容映射到 `WORD_IMAGE_CLICK`。

### UX-007 乱序拼图不是真拼图且红框泄露答案

- 验证码类型：`JIGSAW`
- 优先级：P0
- 状态：已自动验收
- 现象：当前更像“点击交换错位区域”，不是完整乱序拼图；画面里还使用红框/高亮标注差异区域。
- 影响：安全风险高。红框等于直接提示脚本和用户答案区域，防护价值不足。
- 期望：去掉红框、高亮框、答案边界等泄露提示，重做为真正乱序拼图。
- 验收点：
  - 页面不再出现答案提示框或稳定高亮边界。
  - 用户需要根据图片内容判断拼图块位置。
  - 错误交换不能通过，正确还原后通过。
- 当前进展：内置 `JIGSAW` 已改为 2x2/3x3 图片块随机交换，移除红框/答案高亮；Runtime 支持拖动一块到另一块或点击两块交换，并会在界面上直接交换拼图瓦片；拖拽松手自动验证，点击交换需要手动提交，服务端按随机交换瓦片区域做顺序无关校验。

### UX-009 新增图片九宫格点选验证码

- 新增验证码类型：`GRID_IMAGE_CLICK`
- 优先级：P1
- 状态：已自动验收
- 目标形态：类似 Google/reCAPTCHA 的图片格子二次挑战，提示用户选择包含某类目标的图片格子。
- 交互期望：展示 3x3 或 4x4 图片格子，例如“请选择包含交通灯/汽车/桥/楼梯的图片”；用户点选多个格子后提交。
- 校验期望：服务端生成本次目标集合，前端只提交选中格子索引，服务端比对集合。
- 安全约束：不能在前端暴露答案索引；素材和目标类别需要可配置，后续可接资源库或对象标注数据。
- 体验约束：格子清晰、选中态明显但不泄露答案、移动端可点、提示文案明确。
- 验收点：
  - 新类型进入类型矩阵和 Demo。
  - 支持多选格子提交。
  - 错选/漏选失败，正确集合通过。
  - Browser smoke 覆盖正确点选通过和错误点选失败。
- 当前进展：已新增 `GRID_IMAGE_CLICK`，Demo 展示为 3x3 图片格子二次挑战；服务端生成目标格集合并做顺序无关校验，browser smoke 覆盖正确点选通过和错误点选失败。
- 2026-06-23 资源补充：后端资源模型新增 `grid_category_library`，图片格子按 category 管理目标/非目标图库；Runtime 点选后不自动验证，支持再次点击已选格子取消选择。
- 2026-06-24 进展：`grid_category_library` 已接入服务端合成链路，每条资源可通过 metadata 的 `category`/`label` 声明分类；服务端按 session 答案格抽目标分类图片，其它格抽干扰分类图片，不下发目标格索引。

### UX-010 图库资源模型不能停留在单张内置图

- 验证码类型：全部图片型验证码
- 优先级：P0
- 状态：部分完成，背景图库和图片格子分类图库已接入生成链路
- 现象：如果后端长期只依赖内置单图或少量固定背景，脚本可以围绕固定素材建立模板。
- 期望：后端建立图库模型；图片格子使用分类图库，图标点选使用内置/可扩展 SVG 图标库，其它图片型验证码按背景形态建立背景图库。
- 当前进展：资源类型已增加 `background_library`、`concat_background_library`、`jigsaw_background_library`、`grid_category_library`、`icon_library`；管理端资源类型选择器和契约检查已同步。`background_library` 已支持同作用域多资源保留并在服务端合成时随机抽样；`CONCAT` 和 `JIGSAW` 已拆到专用背景图库，不再复用通用背景，便于按图片连续性、切片可辨识度和用户通过难度筛选素材；`grid_category_library` 已支持按分类生成图片格子挑战；管理端资源表单支持 `category`、`label`、`weight` 和扩展 metadata JSON。`icon_library` 仍是登记能力，外部 SVG 图标图库接入渲染链路待补。
