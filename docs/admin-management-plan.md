# 后台管理系统规划

本文档规划 CaptCha 后台管理系统的产品边界、信息架构、核心工作流和落地阶段。后台不应只是管理 API 的表单集合，而应成为验证码平台的运营控制台：帮助使用者完成接入、防护、排障、素材维护、样本复核、模型上线和审计追踪。

## 1. 后台定位

后台面向私有化部署和平台运营场景，核心目标是让管理员能安全、可解释地控制整个人机验证平台。

CaptCha 是平台，不拥有接入方的用户体系。后台和策略中的“账号”“设备”“主体”都只是接入方传入的外部标识或脱敏标识，例如 `account_id_hash`、`device_id_hash`、匿名访客标识、IP 摘要和 User-Agent 摘要。平台不能假设存在平台内用户表，也不保存明文业务用户数据。

后台要解决五类问题：

- 接入配置：创建应用、生成密钥、选择接入方式、确认运行环境。
- 防护策略：配置哪些请求需要验证、限流、阻断、观察或升级验证。
- 验证资源：维护各验证码类型需要的图片、图标、模板、字体和分类素材。
- 风控闭环：查看请求结果、复核样本、导出训练数据、登记和启停模型。
- 系统治理：管理令牌、权限、审计、指标、告警和生产安全状态。

后台不应把底层字段作为主要语言。运营人员看到的应是“应用、路径、验证方式、触发条件、素材分组、风险分、通过率、拦截率、近期变更”，而不是 `client_id`、`resource_type`、`model_mode`、`nonce`、`hash` 等工程字段。

## 2. 平台角色

第一阶段可继续使用轻量管理令牌。后续可以在开源版保持单令牌模式，同时为生产部署预留 RBAC。

推荐角色：

| 角色 | 主要职责 | 关键权限 |
|---|---|---|
| 平台管理员 | 管理整个平台、应用、令牌和系统配置 | 全部读写、密钥轮换、危险操作确认 |
| 接入工程师 | 创建应用、读取接入配置、联调 Gateway / middleware | 应用读写、密钥生成、策略只读或有限编辑 |
| 风控运营 | 管理路由策略、IP 名单、调用 dry-run 接口和审计排查 | 策略读写、审计读取、dry-run 执行 |
| 素材运营 | 上传、启停、删除验证码素材，查看素材健康 | 资源图库读写、素材指标读取 |
| 模型运营 | 样本复核、数据导出、模型登记、启用和回滚 | 样本读写、模型读写、指标读取 |
| 审计员 | 查看审计和指标，不做配置变更 | 只读 |

## 3. 信息架构

当前后台已经具备概览、应用、路由策略、IP 策略、策略 dry-run API、资源图库、审计、样本复核和模型管理。后续建议按工作域组织，而不是简单按数据表组织。

推荐导航结构：

```text
工作台
  概览
  告警与健康

接入中心
  应用
  密钥与接入
  Gateway / 中间件示例

策略中心
  路由策略
  IP 名单
  策略版本

开发者接口
  接入配置
  策略 dry-run API
  Gateway / 中间件示例

资源中心
  资源图库
  素材健康
  上传任务

风控与模型
  样本复核
  数据导出
  模型管理
  模型评估

审计与系统
  审计日志
  管理令牌
  权限角色
  系统配置
```

当前一期可以保持现有顶层导航，但目标态不把策略模拟做成常驻页面。策略模拟应作为 dry-run API 提供给接入检查、自动化测试、CLI、业务后台或外部运维工具调用；管理台最多展示接口说明、请求示例、最近调用结果和接入检查状态。

## 4. 核心工作流

### 4.1 首次接入

首次接入要让接入方管理员从空后台走到可用防护链路。

流程：

1. 新增应用，填写应用名称和应用标识。
2. 生成应用密钥，只展示一次明文。
3. 选择接入方式：iframe、Gateway、多语言 middleware、gRPC。
4. 自动生成当前应用的接入示例配置。
5. 创建默认路由策略，例如登录、注册、短信发送、下单。
6. 上传或确认默认素材库。
7. 调用策略 dry-run API 检查路径是否命中策略。
8. 在概览中确认近期验证请求、通过率和审计事件。

后台应把这条链路做成“接入检查清单”，避免接入方只看到一堆空表。

### 4.2 风险处置

当某个业务接口出现异常流量时，后台要支持快速判断和回滚。

流程：

1. 在概览或告警中看到拦截率、失败率、验证请求异常。
2. 进入审计，按应用、场景、路径、原因、外部账号标识、设备标识或 IP 筛选。
3. 调用策略 dry-run API 复现请求上下文。
4. 调整路由策略、限流策略、IP 名单或验证方式。
5. 先用灰度生效范围发布。
6. 观察通过率、拦截率和误伤反馈。
7. 必要时回滚到上一策略版本。

### 4.3 素材运营

验证码素材会影响通过率、失败率和攻击成本。资源图库应按“可用素材资产”而不是“资源配置表”来设计。

流程：

1. 选择应用和图库类型。
2. 上传图片、图标或 ZIP。
3. 后端校验 MIME、尺寸、大小、URI、checksum 和安全来源。
4. 素材进入启用或待处理状态。
5. 概览和素材健康显示按验证码类型、素材类型、素材分组聚合的命中量和失败率。
6. 对高失败率素材执行停用、替换或删除。

后续可增加自动体检：大面积纯色、尺寸过小、主体偏移、重复纹理、分类缺失等。

### 4.4 样本与模型迭代

模型链路应是离线训练和受控上线，不做请求路径上的在线学习。

流程：

1. 验证结果异步写入风险特征快照。
2. 样本复核页按应用、场景、验证码类型、标签和入训状态筛选。
3. 人工标注明确真人或机器样本。
4. 导出训练 JSONL。
5. 离线训练生成模型 artifact。
6. 在后台登记模型名称、版本、样本版本、训练窗口、模型文件和质量指标。
7. 先以影子或观察模式启用。
8. 指标稳定后切到执行模式。
9. 出现误伤时一键回滚上一版。

## 5. 模块规划

### 5.1 概览

概览要回答“平台现在是否健康”。

一期指标：

- 验证通过率、验证请求、拦截请求、启用素材。
- 应用、路由策略、IP 策略、近期配置变更。
- 素材命中和失败率 Top 列表。
- 入训样本、真人样本、机器样本、启用模型。
- 高频场景和常见原因。

后续增强：

- 时间范围选择。
- 趋势图和环比。
- 按应用、场景、路径下钻。
- 告警入口：通过率骤降、拦截率异常、素材失败率异常、模型误伤风险。

### 5.2 应用

应用是后台的租户和接入边界。

一期能力：

- 新增和编辑应用。
- 启用、停用应用。
- 生成和轮换应用密钥。
- 显示密钥是否已配置，不展示明文。
- 设置默认异常处理策略。

后续增强：

- 接入指南和配置片段。
- 应用健康状态。
- 应用级默认场景策略。
- 应用配置导入导出。
- 禁用应用前的影响面提示。

### 5.3 策略中心

策略中心是后台最核心的操作面，必须强调安全发布和可解释。

一期能力：

- 全站防护策略：不强制绑定具体路径或方法，适合作为接入站点的默认保护。
- 接口级覆盖策略：按应用、路径、方法、场景、外部主体上下文、风险信号和频率信号组合条件，允许短信发送、发券、提现等接口配置永远验证并覆盖全站通行态。
- 内置策略模板：无通行态时验证、永远验证、访问过快、风险较高、可信外部主体低风险跳过验证；模板只是快速创建策略，不限制底层策略能力。
- IP 名单：放行名单、拦截名单。
- 策略 dry-run API：不创建 session、不消费 ticket、不写审计，返回命中策略、决策动作、原因、验证码类型和关键风险摘要。
- 灰度生效范围。
- 失败后升级验证。

后续增强：

- 策略版本：草稿、发布、历史版本、回滚。
- 冲突检测：路径重叠、优先级冲突、灰度覆盖、名单覆盖。
- 策略变更 diff。
- 发布前模拟：批量输入路径样本，检查命中结果。
- 常用模板：登录防护、注册防护、短信防护、支付防护。

策略不能固化为少数几种模式。后台可以把常见策略包装成模板，但底层模型应由这些部分组成：

```text
scope
  应用、场景、路径、方法、生效时间、生效范围；路径和方法可留空表示全应用。

conditions
  all / any / not 组合条件。
  可引用 IP、外部账号 hash、设备 hash、匿名访客 hash、UA 摘要、请求频率、历史验证状态、clearance、外部风险分、模型分、素材分组等信号。

aggregation
  计数窗口、计数维度、固定窗口 / 滑动窗口 / 令牌桶、冷却时间。

actions
  allow、skip_challenge、challenge、step_up_challenge、rate_limit、cooldown、block、observe、require_business_verify。

governance
  优先级、灰度、版本、草稿、发布、回滚、审计和 dry-run。
```

策略 dry-run 不作为后台主页面。它应是稳定接口能力，用于接入检查、CI 回归、业务系统联调和运维排障。后台可以在应用详情或接入中心提供 curl 示例、请求 schema、响应说明和最近 dry-run 记录，但不把它设计为独立运营工作台。

### 5.4 资源图库

资源图库要面向素材运营，而不是工程配置。

一期能力：

- 上传图片或 ZIP。
- 按通用背景、滑动还原、乱序拼图、旋转校准、图片格子、图标图库分类。
- 按应用过滤。
- 多选启停和删除。
- 图片预览。
- 素材分组空值和 `default` 显示为“通用”。

后续增强：

- 上传任务状态和失败原因。
- 素材质量评分。
- 分类标签批量编辑。
- 按验证码类型查看素材覆盖率。
- 按失败率推荐停用素材。
- 隔离待审素材和线上素材。

### 5.5 审计

审计要支持追责、排障和配置复盘。

一期能力：

- 按应用、场景、动作、结果、原因、外部账号标识、设备标识筛选。
- 原因以中文运营文案展示，原始码仅作辅助。
- 管理 API 的配置变更写入审计事件。

后续增强：

- 配置变更 diff。
- 操作者身份。
- 请求时间线：策略命中、验证码创建、验证结果、ticket 消费、Gateway 降级事件。
- 导出审计结果。
- 危险操作二次确认记录。

### 5.6 样本复核

样本复核是人工反馈入口，要避免把特征 JSON 当成主界面。

一期能力：

- 按应用、场景、验证码类型、标签、入训状态筛选。
- 标记真人、机器、未知。
- 控制是否进入训练集。
- 导出训练数据。

后续增强：

- 样本详情抽屉：轨迹摘要、风险分、资源摘要、失败原因。
- 批量标注。
- 复核队列。
- 标注一致性和复核人记录。
- 训练集版本冻结。

### 5.7 模型管理

模型管理要表达“上线方式”和“质量”，而不是只展示 artifact 字段。

一期能力：

- 登记模型名称、版本、特征版本、训练窗口、模型文件、指标。
- 启用模型。
- 回滚模型。
- 区分影子、观察、执行模式。

后续增强：

- 模型评估报告：区分度、召回率、误伤率、样本覆盖。
- 与训练集版本关联。
- 模型灰度。
- 上线审批。
- 模型效果趋势。

### 5.8 系统与权限

开源版可以保持简单，但生产部署需要清晰边界。

一期能力：

- 管理令牌检查。
- 令牌本机保存和清除。
- 401 后要求重新输入。
- 生产安全闸门通过环境变量强制关键配置。

后续增强：

- 多管理员账号或外部 OIDC。
- 角色权限。
- 管理操作审计主体。
- API token 管理。
- mTLS / 反向代理身份透传。
- 安全配置体检页。

## 6. 数据和接口规划

当前后台 API 已覆盖主要闭环：

- `/api/v1/admin/applications`
- `/api/v1/admin/route-policies`
- `/api/v1/admin/policies`
- `/api/v1/admin/ip-policies`
- `/api/v1/admin/policy/simulate`
- `/api/v1/admin/resources`
- `/api/v1/admin/resources/upload`
- `/api/v1/admin/audit-events`
- `/api/v1/admin/risk-feature-snapshots`
- `/api/v1/admin/risk-model-versions`
- `/api/v1/admin/metrics`

后续建议新增或扩展：

| 能力 | 接口方向 | 说明 |
|---|---|---|
| 策略规则发布治理 | `/api/v1/admin/policies/{id}/publish` / `/rollback` | 在已落地规则 CRUD 上补草稿、发布、回滚和 diff |

| 权限角色 | `/api/v1/admin/admin-identities` / `/roles` | 生产部署可选；这里指平台管理身份，不是接入方业务用户 |

## 7. 体验原则

后台体验应遵循这些原则：

- 应用名称优先，应用标识只作为辅助信息。
- 主操作按钮必须说明对象，例如“新增应用”“保存策略”“上传图片”。
- 页面文案使用运营语言，工程字段只在详情、hover 或高级区域出现。
- 危险操作必须确认，密钥明文只展示一次。
- 策略编辑根据触发条件动态展示字段，不让无关配置互相干扰。
- 策略 dry-run API 响应要突出决策、原因、验证码、命中路由和风险摘要，避免暴露内部副作用或调试噪声。
- 资源图库以文件和素材健康为主，不默认展示底层 URI。
- 概览要能在 30 秒内判断平台状态。
- 所有列表都要支持应用范围过滤，避免多租户数据混杂。
- 所有配置变更都应可审计、可解释、可回滚。

## 8. 分期路线

### 阶段 1：后台基础闭环

目标：把现有能力整理成稳定可用的一期后台。

任务：

- 收敛后台主导航，策略模拟不再作为目标态常驻页面。
- 完善空状态、加载状态、错误状态。
- 继续强化中文运营文案。
- 拆分 `web/admin/src/main.tsx`，按页面、API hooks、常量、工具函数组织。
- 补齐管理台浏览器 smoke，覆盖令牌、应用、资源图库主流程；策略 dry-run 走 API 契约测试。
- 为当前 API 和页面增加后台规划文档链接。

验收：

- 新接入方能完成应用创建、密钥生成、路由策略配置、dry-run API 校验、资源上传和指标查看。
- 常用操作不需要理解底层字段。

### 阶段 2：配置安全和发布治理

目标：降低策略误配和线上变更风险。

任务：

- 策略草稿、发布、回滚。
- 策略冲突检测。
- 配置变更 diff。
- 危险操作审计主体。
- 应用禁用、密钥轮换、批量删除的影响面提示。
- 配置导出导入。

验收：

- 所有关键配置变更可审计、可回滚。
- 策略发布前能通过 dry-run API 和冲突检测发现常见问题。

### 阶段 3：运营观测和告警

目标：让后台从“配置台”升级为“运营台”。

任务：

- 时间范围和趋势图。
- 按应用、场景、路径下钻。
- 素材失败率告警。
- 通过率和拦截率异常提示。
- 审计时间线。
- Prometheus 指标和后台指标对齐。

验收：

- 运营人员能从概览定位到具体策略、素材或场景。

### 阶段 4：风控模型运营

目标：形成可控的模型迭代闭环。

任务：

- 样本详情和批量标注。
- 训练集版本。
- 模型评估报告。
- 模型灰度和审批。
- 模型效果趋势。

验收：

- 模型可以从样本复核、离线训练、登记、影子、观察、执行到回滚全流程管理。

### 阶段 5：生产权限和多团队协作

目标：支持更真实的企业部署。

任务：

- OIDC / SSO 可选接入。
- RBAC。
